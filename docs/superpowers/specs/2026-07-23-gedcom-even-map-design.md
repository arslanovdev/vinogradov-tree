# Поддержка событий GEDCOM `EVEN` в карточках и на карте

Дата: 2026-07-23

## Контекст

В `public/fedorovka_family.ged` военные эпизоды можно описывать стандартным
GEDCOM-событием `EVEN` с дочерними полями `TYPE`, `DATE`, `PLAC` и
`MAP/LATI/LONG`. Например, у Тимофея Романовича Камышлова уже зафиксирован
подтверждённый наградным документом бой 17 июля 1943 года у села Русское.

Текущий парсер сохраняет только `BIRT`, `DEAT`, `BURI` и `RESI`, поэтому
`EVEN` теряется: событие не отображается ни в карточке человека, ни на карте.

## Цель

Добавить полноценное чтение стандартных событий `EVEN`, чтобы подтверждённые
военные и другие значимые эпизоды:

- показывались среди фактов в карточке человека;
- становились маркерами карты при наличии координат;
- учитывались в списке мест без координат;
- не требовали нестандартных тегов или подмены события местом жительства.

## Не входит в задачу

- Автоматическое построение полного боевого маршрута.
- Дорисовывание линий между местами, если последовательность перемещений не
  подтверждена источниками.
- Геокодирование мест по названию и добавление предположительных координат.
- Редизайн карты или карточки человека.
- Поддержка всех возможных GEDCOM-событий и атрибутов за пределами `EVEN`.

## Модель данных

В `Indi` добавляется поле:

```ts
events: GEvent[];
```

Оно всегда инициализируется пустым массивом. Каждый элемент соответствует
одной записи верхнего уровня `1 EVEN`. Для события используются уже
существующие поля `GEvent`:

- `type` — значение дочернего `2 TYPE`;
- `date` — значение `2 DATE`;
- `plac` — значение `2 PLAC`;
- `lat` и `lon` — значения `4 LATI` и `4 LONG` внутри `3 MAP`.

Порядок событий сохраняется таким же, как в GEDCOM. Это важно для событий с
неполными датами и для предсказуемого отображения.

## Парсинг

При встрече `1 EVEN` парсер:

1. создаёт новый `GEvent`;
2. добавляет его в `currentIndi.events`;
3. назначает его текущим событием;
4. передаёт дочерние `TYPE`, `DATE`, `PLAC`, `MAP`, `LATI`, `LONG` в уже
   существующую логику разбора событий.

Начало любого следующего события или другого поля первого уровня завершает
предыдущее событие тем же естественным образом, как сейчас для `BIRT`,
`DEAT`, `BURI` и `RESI`.

Пустые или частичные события допустимы: парсер не падает, а интерфейс
показывает только имеющиеся поля. Событие без содержательных полей не
выводится в карточке и не попадает на карту.

## Единое представление событий

Чтобы не расширять хрупкую логику с числовыми индексами, в модели мест
формируется явный список описаний событий человека. Каждый элемент содержит:

- само событие;
- человекочитаемую подпись;
- категорию места для карты.

В список входят четыре существующих события и все `Indi.events`.

Подписи для стандартных событий остаются прежними:

- `Рождение`;
- `Смерть` или уточнённый тип смерти;
- `Захоронение`;
- `Жительство`.

Для `EVEN` используется `TYPE`, а при его отсутствии — `Событие`.

## Отображение в карточке

Содержательные `EVEN` добавляются в `Detail.facts` после основных
биографических фактов, в порядке GEDCOM.

- Заголовок факта: `TYPE` либо `Событие`.
- Значение: дата и место, разделённые тем же знаком `·`, который уже
  используется в карточке.
- Если `TYPE` есть, но нет даты и места, событие не дублируется в значении:
  показывается только тогда, когда у него есть дополнительное содержимое.

Пример:

```text
Боевой эпизод
17.07.1943 · с. Русское, Ростовская область
```

## Отображение на карте

Событие попадает на карту только при одновременном наличии:

- непустого `PLAC`;
- конечных числовых `lat` и `lon`.

Маркер объединяется с существующим маркером того же нормализованного места,
если координаты и название совпадают по действующим правилам.

Категория `war` назначается событию, если его `TYPE` содержит одно из
устойчивых военных обозначений: бой, боевой эпизод, служба, ранение, плен,
освобождение из плена, мобилизация или переправа. Сопоставление
регистронезависимое. Все прочие `EVEN` используют обычную классификацию места.

В панели места показываются подпись события и дата. Автоматическая линия от
дома к месту боя не строится: одна подтверждённая точка не доказывает
маршрут перемещения.

Событие с `PLAC`, но без координат, учитывается в
`placesMissingCoords`, чтобы исследователь видел, какое место ещё нужно
локализовать.

## Обратная совместимость

- Люди без `EVEN` отображаются без изменений.
- Существующие `BIRT`, `DEAT`, `BURI`, `RESI`, маркеры, кластеры и линии
  продолжают работать.
- Старые вызовы, создающие тестовые `Indi`, должны либо передавать
  `events: []`, либо тип допускает безопасное отсутствие поля только на
  период миграции. Предпочтительный итоговый контракт — обязательный массив.
- Источник истины остаётся GEDCOM; отдельное приложение-хранилище событий не
  создаётся.

## Проверка

### Модульные тесты

1. Парсер читает несколько `EVEN` у одного человека и сохраняет `TYPE`,
   `DATE`, `PLAC`, `LATI`, `LONG`.
2. `mapPlaces` создаёт военный маркер для `EVEN` с координатами и сохраняет
   правильную подпись и дату.
3. `placesMissingCoords` включает место события без координат.
4. `buildDetail` добавляет событие в факты карточки.
5. Событие без даты, места и дополнительного содержания не создаёт пустой
   факт или маркер.
6. Существующие тесты карты и структуры GEDCOM продолжают проходить.

### Проверка на реальных данных

- В карточке Тимофея Романовича Камышлова отображается бой 17.07.1943 у
  села Русское.
- На карте появляется Русское с категорией фронтового места и тем же
  событием в панели.
- Между Фёдоровкой и Русским не появляется неподтверждённая линия маршрута.

### Общая проверка репозитория

После реализации выполняются:

```sh
npm test
npm run build
```

Дополнительно проверяются целостность ссылок GEDCOM и наличие всех файлов,
указанных в `OBJE/FILE`.

## Критерии готовности

Задача завершена, когда стандартный `EVEN` сохраняется парсером, виден в
карточке, участвует в карте и отчёте об отсутствующих координатах, а все
тесты и production-сборка проходят без новых ошибок.
